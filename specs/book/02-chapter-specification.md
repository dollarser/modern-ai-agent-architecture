# 章节规范

## 共同必备项

每章必须有：明确标题、学习目标、前置知识或适用读者、核心正文、来源类型或引用、章节 Checklist。凡包含时效性产品或协议信息，必须给出核查版本或日期。

## 核心知识点的决策叙事

对会影响架构、实现或选型的**核心知识点**，正文应让读者能依次回答以下五个问题；可采用小节、对比表、流程图或案例组合表达，不要求为每个术语机械套用五个小标题。

| 问题 | 必须讲清的内容 | 常见缺口 |
|------|----------------|----------|
| Why | 它解决什么真实约束、失败模式或业务目标？ | 只有定义，读者不知道为何需要它 |
| What | 概念边界、责任归属，以及与相邻概念的区别是什么？ | 把协议、产品功能和实现模式混为一谈 |
| How | 最小实现路径、关键步骤、接口或运行机制是什么？ | 只有口号，或只有无法迁移的代码 |
| When | 适用信号、前置条件、规模阈值和不适用情形是什么？ | 把一种模式写成默认答案 |
| Trade-off | 成本、延迟、复杂度、可靠性、安全性和可观测性的代价是什么？如何缓解？ | 只列优点，不说明失败代价 |

概念导读章可把五个问题分散在背景、术语边界和案例中；实现章至少应在代码前给出 Why / What，并在代码后给出 When / Trade-off；协议与产品对比章还必须区分规范事实、公开证据和工程推导。

## 按类型适用的内容

| 章节类型 | 额外要求 |
|------|------|
| 概念 / 历史 | 背景、关系图或时间线、术语边界、至少一个可复核例子 |
| 实现 / 工程 | 架构图、完整或可定位的运行示例、依赖与预期行为、失败与安全边界 |
| 协议 | 版本范围、角色与生命周期、标准字段和教学模拟的边界、官方规范链接 |
| 产品 / 框架对比 | 比较维度、证据类型、核查日期、能力变化风险与选型限制 |
| FAQ / 汇总 | 指向权威章节，避免复制整段正文；回答必须可追溯 |

## 可选而非机械要求

Mermaid 图、FAQ、最佳实践、反模式和可运行代码应在能提升理解时采用。所有章节强制塞入这些栏目会造成重复和伪示例；缺少时应在 Checklist 中说明替代材料或不适用原因。
