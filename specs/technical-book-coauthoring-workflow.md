# 与 AI 协作编写完整技术书：从需求到出版冻结的可复用工作流

> **用途：** 将一个感兴趣的技术主题，逐步发展为概念准确、结构完整、代码可运行、事实可追溯、适合持续维护和公开发布的技术书。
> **适用范围：** 架构、编程语言、框架、协议、基础设施、数据工程、安全、AI 等需要同时讲解理论和实践的主题。
> **核心原则：** 先建立规范，再写内容；先形成闭环，再扩展覆盖；先审查事实和结构，再润色文字；最终用实际书稿反向修订规范。

---

## 1. 为什么需要分阶段共创

直接要求 AI“一次生成一本书”通常会产生以下问题：

- 章节看似完整，但核心概念的边界前后不一致；
- 内容覆盖面很广，却没有清晰的学习路径和最终成果；
- 代码、图表、正文分别合理，组合后却不匹配；
- 使用大量“完整实现”“生产级”“最佳实践”等无法验证的标签；
- 产品和协议事实很快过时，且没有来源与核查日期；
- 后续修订只改正文，没有同步目录、术语表、索引和需求文档；
- 多轮重构后，旧 PRD 反而会覆盖已经更合理的新架构。

因此，技术书共创应采用迭代式工程流程：

```text
主题与读者
→ 初始需求文档
→ 需求升级为可执行规范
→ 书籍骨架与最小闭环
→ 分章正式写作
→ 自动化基线
→ 多轮专项审查
→ 纵向架构审查
→ 细节与出版修订
→ 反向更新需求文档
→ 候选版本冻结
```

每个阶段都有独立产物和退出条件。未通过前一阶段的验收，不应靠继续扩写掩盖问题。

## 2. 阶段一：明确主题、读者和最终学习成果

### 2.1 先回答六个问题

1. 这本书主要解决什么学习或工程问题？
2. 读者已有多少背景知识？
3. 读者读完后应该能够解释什么？
4. 读者读完后应该能够实现什么？
5. 哪些内容明确不属于本书范围？
6. 最终用什么可验证成果证明“读完了”？

不要只写“全面介绍某技术”。应把目标改写为可以验收的句子，例如：

> 本书从核心概念和理论出发，逐步实现一个可运行的教学系统；读者最终能够完成某类真实但受限的任务，并理解它与生产系统之间的差距。

### 2.2 定义三个内容层次

| 层次 | 回答的问题 | 常见产物 |
|---|---|---|
| 概念与理论 | 为什么需要、是什么、边界在哪里 | 定义、关系图、历史背景、论文或规范 |
| 架构与实现 | 系统如何组成、关键接口是什么 | 架构图、数据结构、状态机、代码示例 |
| 使用与治理 | 何时采用、如何验证、风险是什么 | 案例、测试、安全控制、部署与评估 |

如果一本技术书只有第一层，它更像概念手册；只有第二层，它更像代码教程；只有第三层，它容易变成经验列表。完整技术书需要三者形成闭环。

### 2.3 本阶段产物

- 一段不超过 150 字的书籍定位；
- 目标读者清单；
- 5～10 项最终学习成果；
- 明确的非目标；
- 一个最小可运行或可验证的最终案例。

### 2.4 可复用提示词

```text
我希望围绕【主题】写一本完整技术书，目标读者是【读者背景】。
请先不要写章节正文，先帮我明确：
1. 书籍定位；
2. 读者读完后应掌握的概念；
3. 读者应能实现的系统或任务；
4. 本书明确不覆盖的范围；
5. 可以自动或人工验证的最终成果。
请把目标写成可验收要求，不要使用“全面、深入、完整”等无法验证的形容词。
```

### 2.5 当作者了解领域但不完全熟悉时，先管理“不知道”

对一个领域有使用经验，并不等于熟悉它的完整知识结构。此时不要立即让 AI 生成“完整目录”，而应先建立认知边界表：

| 分类 | 含义 | 处理方式 |
|---|---|---|
| 已知且可验证 | 自己理解，也能找到规范、源码、实验或案例支持 | 可以进入初始主线 |
| 已知但需复核 | 熟悉名词和常见用法，但不确定边界、版本或内部机制 | 标记为待调研，不直接写成 Fact |
| 推测相关 | 感觉重要，但尚不知道是否应独立成章 | 先放 Coverage Backlog，不承诺章节 |
| 已知未知 | 明确知道自己缺少的部分 | 安排资料审查、原型或专家反馈 |
| 潜在未知 | 当前目录之外可能存在的重要主题 | 用生命周期、失败模式和参考实现交叉扫描 |

这里的目标不是列出所有未知，而是避免未知内容在写作中被 AI 自动补成听起来合理的断言。

可以为重要主张增加置信度：

```text
High：当前规范、官方源码或可重复实验直接支持
Medium：多个一手来源或成熟实现一致，但不存在统一规范
Low：单一实现、快速变化产品或工程推导
Unknown：尚未完成资料核查
```

`Low/Unknown` 可以进入调研清单和草稿，但不能作为冻结后的核心架构前提。

### 2.6 用“证据扫描”建立领域地图

作者不完全熟悉领域时，章节规划前应完成一次短周期 Discovery Sprint。不要先搜“这个领域有哪些知识点”，而是从五类一手对象反推知识结构：

1. **标准与规范：** 领域正式定义了哪些角色、对象、状态和兼容边界？
2. **官方文档：** 主流产品把哪些能力暴露给用户？哪些只是产品术语？
3. **代表性开源实现：** 真实系统如何拆模块、处理生命周期和失败？
4. **原始论文或经典资料：** 哪些理论是后续实现的来源？适用假设是什么？
5. **真实任务与事故：** 用户到底完成什么工作？常见失败、安全和运维问题是什么？

扫描时使用统一表格，而不是收藏一堆链接：

| 候选概念 | 解决的问题 | 证据来源 | 是否跨实现通用 | 与相邻概念的边界 | 暂定去向 |
|---|---|---|---|---|---|
| 概念 A | 失败模式或目标 | 规范/源码/论文 | 高/中/低 | A ≠ B | 主章节/案例/附录/删除 |

至少选择三种不同性质的参考对象，例如“一个规范、一个成熟开源实现、一个主流产品”。只参考单一框架，容易把其目录结构误认为整个领域的知识结构。

Discovery Sprint 的退出条件不是“资料看完”，而是：

- 能画出主要对象、生命周期和依赖关系；
- 能解释三组以上容易混淆的概念；
- 能列出代表性任务和失败模式；
- 能指出哪些内容是标准、共同模式、单一产品设计或个人推导；
- 能说出哪些候选主题不值得进入本书。

### 2.7 用成果阶梯定义可验收目标

对不完全熟悉的领域，最稳妥的做法不是承诺“精通”，而是为每类内容选择成果层级：

| 层级 | 读者最终能够 | 可验收证据 |
|---|---|---|
| L1 识别 | 识别术语、对象和典型场景 | 术语边界题、概念关系图 |
| L2 解释 | 解释机制、生命周期和设计原因 | 用自己的话复述、状态或时序分析 |
| L3 比较 | 根据约束比较多种方案 | 带反例和 Trade-off 的决策表 |
| L4 实现 | 构建最小但真实的纵向切片 | 可运行代码、测试、失败路径 |
| L5 运用 | 在代表性任务中组合多个模块 | 端到端案例、验收场景 |
| L6 评估 | 诊断、评估和改进系统 | 评估集、轨迹、性能或安全检查 |
| L7 生产治理 | 在真实组织中可靠运营 | SLO、迁移、审计、灾备和运维证据 |

一本面向开发者的完整技术书通常应达到：

- 核心概念达到 L2～L3；
- 关键开发主线达到 L4～L5；
- 测试、安全和评估达到 L5～L6；
- L7 可以说明生产要求和差距，但除非书中真的提供基础设施，不应宣称已经实现。

这能防止作者因为不熟悉生产环境，就在教学示例上贴“生产级”标签；也避免为了显得完整，强行对每个边缘概念提供代码。

### 2.8 把可验收成果写成行为契约

成果应描述读者或系统能观察到的行为，而不是使用抽象形容词。

不推荐：

```text
深入理解分布式系统。
掌握完整安全体系。
实现生产级框架。
```

推荐：

```text
读者能画出请求从客户端到三个服务的调用和失败传播路径。
读者能解释重试、超时和幂等键为什么不能互相替代。
示例能在单节点故障后返回结构化错误，并证明不会重复提交副作用。
读者能根据延迟、吞吐、一致性和运维成本选择两种候选设计之一。
```

一个完整的验收契约包含：

```text
Given：给定的背景、数据、代码或约束
When：读者执行的操作、推理或实现任务
Then：可以观察和核对的结果
Failure：至少一个必须正确处理的失败情况
Boundary：本成果明确不证明什么
Evidence：测试、图表、引用、运行输出或评估方式
```

示例：

```text
Given：一个包含两个服务和一个持久化存储的教学项目
When：读者加入有限重试并模拟下游超时
Then：请求在总预算内终止，日志能关联同一请求的所有尝试
Failure：非幂等写入不得在未知提交状态下自动重试
Boundary：不声称具备跨地域高可用
Evidence：自动测试、Trace 输出和状态图
```

如果暂时无法写出 `Then/Failure/Evidence`，说明这个学习成果尚未足够明确，不应据此冻结章节。

### 2.9 用“代表性任务集”校验成果是否遗漏

只定义一个顺利完成的 Happy Path，无法证明书籍覆盖完整。应建立一个小型代表性任务集：

| 任务类型 | 目的 |
|---|---|
| 最小成功任务 | 验证核心闭环 |
| 常见业务任务 | 验证组合能力和真实价值 |
| 错误输入 | 验证校验与错误信息 |
| 下游失败 | 验证超时、重试、回退和终态 |
| 权限或安全拒绝 | 验证系统不会越权 |
| 版本或配置变化 | 验证兼容、迁移或恢复边界 |
| 不适用场景 | 证明读者知道何时不该使用本方案 |

任务集不必很大。5～8 个经过选择的任务，通常比 30 个相似示例更能暴露知识结构缺口。

### 2.10 从依赖图生成章节，而不是从术语表生成章节

先画知识和能力依赖：

```text
基础对象
→ 最小机制
→ 可运行闭环
→ 状态与失败
→ 扩展和组合
→ 安全与治理
→ 生产差距与评估
```

然后把依赖图压缩成章节。一个候选主题只有满足以下至少一项时才值得独立成章：

- 是三个以上后续主题的前置知识；
- 拥有独立生命周期、状态机或安全边界；
- 需要一套独立实现和验证；
- 与相邻概念存在高频混淆，合并后会损害理解；
- 是最终案例中的关键可替换模块。

以下情况通常不应独立成章：

- 只是另一个概念的一个配置字段；
- 只有单一产品使用该术语；
- 没有独立决策、实现或失败模式；
- 内容只能重复前章定义；
- 作者目前只能找到营销材料，无法建立可靠证据。

### 2.11 为不确定章节设置“暂定状态”

初始目录可以包含不同成熟度：

```text
Committed：核心主线，证据和验收成果明确
Provisional：大概率需要，但边界或材料尚未稳定
Backlog：可能相关，等待 Discovery Sprint
Appendix Candidate：有参考价值，但不承担主线
Rejected：已判断不适合本书，并记录原因
```

在正式写作前，只冻结 `Committed`。`Provisional` 必须通过证据扫描和代表性任务验证后才能升级。这样可以避免 AI 把每个候选名词都扩写成一章，也允许作者在学习过程中调整认知。

### 2.12 用二维矩阵发现“未知未知”

章节目录回答“按什么顺序学”，但不能单独证明覆盖完整。再用两个维度交叉检查：

**生命周期维度：**

```text
定义/配置 → 创建 → 执行 → 失败 → 恢复 → 更新 → 停止/删除 → 观测
```

**工程质量维度：**

```text
正确性 → 性能 → 可靠性 → 安全 → 可观测性 → 可维护性 → 成本 → 兼容性
```

将核心对象放入矩阵，不要求每个格子都有一章，而是检查是否遗漏关键问题。例如作者熟悉某框架的正常调用，却可能通过矩阵发现自己没有研究升级、撤销、权限拒绝或故障恢复。

### 2.13 不熟悉领域时的章节规划流程

```text
写下已有认知和最终兴趣
→ 建立已知/未知清单
→ 选择规范、开源实现、产品、论文和真实任务
→ 生成候选概念与能力地图
→ 定义 5～10 个行为验收成果
→ 建立 5～8 个代表性任务
→ 画知识与实现依赖图
→ 形成 Committed / Provisional / Backlog 目录
→ 做生命周期 × 工程质量覆盖扫描
→ 只冻结第一阶段和最小纵向闭环
→ 在写作和原型中逐步升级后续章节
```

这种流程允许作者一边写一边成为更熟悉该领域的人，但不会让学习过程失去规范和证据约束。

### 2.14 针对“不完全熟悉领域”的可复用提示词

```text
我对【领域】有一定使用经验，但不熟悉完整理论、标准和生产实践。
请不要直接生成一本书的最终目录，也不要假定我的概念清单完整。

请先完成 Discovery Sprint：
1. 将我目前的认识分为已知可验证、已知待复核、推测相关、已知未知；
2. 从标准/规范、代表性开源实现、主流产品、原始论文和真实失败案例建立领域地图；
3. 区分跨实现通用概念、单一产品术语和工程推导；
4. 提出 5～10 个按 L1～L7 分级的可验收学习成果；
5. 为关键成果写 Given/When/Then/Failure/Boundary/Evidence；
6. 建立代表性任务集和知识依赖图；
7. 给出 Committed、Provisional、Backlog、Appendix Candidate 和 Rejected 的章节候选；
8. 用生命周期 × 正确性/性能/可靠性/安全/可观测性/维护成本矩阵寻找遗漏。

所有关键主张请给出证据类型和置信度。无法验证的内容保留为调研问题，不要补写成事实。
```

## 3. 阶段二：撰写初始需求文档

初始 PRD 的目标不是预测最终所有章节，而是让后续写作有稳定约束。

### 3.1 初始 PRD 应包含什么

```text
Book PRD
├── 定位与读者
├── 内容范围与非目标
├── 核心概念清单
├── 初始章节规划
├── 最终实践案例
├── 写作与术语原则
├── 图表与代码要求
├── 引用与事实验证要求
├── 项目目录与发布方式
└── 验收标准
```

### 3.2 初始章节规划的正确粒度

初始 PRD 可以规划章节职责，但不要提前固定所有小节。推荐先划分学习阶段：

1. 基础认知；
2. 构建最小闭环；
3. 可靠运行；
4. 扩展与互操作；
5. 生产与评估；
6. 案例、FAQ 和索引。

每章只需先回答：

- 这章解决什么问题？
- 读者进入本章前需要知道什么？
- 本章产生什么新能力？
- 下一章为什么自然出现？

### 3.3 初始验收标准不要只写数量

不推荐：

- 至少 50 张图；
- 至少 30 个示例；
- 每章固定 14 个小节；
- 必须达到某个字数。

推荐：

- 核心概念有唯一规范定义；
- 协议和产品事实有直接来源和核查日期；
- 实现章节具有可运行示例或明确的教学片段；
- 正文、图表、代码和 Checklist 一致；
- 构建、链接、图表和测试可自动验证；
- 数量只作为当前快照，不作为扩写目标。

### 3.4 本阶段退出条件

- 章节之间没有明显职责重叠；
- 最终实践案例可以映射到多个章节；
- PRD 明确事实、推导、观点和教学实现的区别；
- 所有量化指标都能说明其质量意义，否则删除。

## 4. 阶段三：把需求升级为可执行 BookSpec

随着讨论深入，初始 PRD 通常会出现版本堆叠：V1、V2、V6、整合版等。如果不治理，AI 和作者会误读旧规范。

### 4.1 PRD 与 BookSpec 的区别

| 文档 | 主要作用 |
|---|---|
| PRD | 记录为什么写、面向谁、覆盖什么 |
| BookSpec | 规定当前如何写、如何审、如何验收和维护 |
| 历史需求 | 保存决策演进，不再约束当前实现 |

### 4.2 建议的 BookSpec 结构

```text
BookSpec/
├── 00-overview.md
├── 01-book-requirements.md
├── 02-chapter-specification.md
├── 03-style-guide.md
├── 04-diagram-guide.md
├── 05-code-guide.md
├── 06-citation-policy.md
├── 07-coverage-matrix.md
├── 08-acceptance-criteria.md
├── 09-review-checklist.md
├── 10-maintenance-guide.md
└── 11-current-baseline.md
```

### 4.3 建立规范优先级

必须显式声明冲突时听谁的：

```text
当前官方规范或源码
→ 已验证的当前实现和实际目录
→ 现行 BookSpec
→ 当前书稿中的时效性声明
→ 历史 PRD
```

这条规则非常重要：后期实现通常会暴露初始需求中的不合理假设，不能让旧 PRD 覆盖已经验证的新结构。

### 4.4 Current Baseline 的作用

集中记录动态快照：

- 当前章节数量和分组；
- 图表数量；
- 示例工程与语言入口；
- 测试数量；
- 当前协议版本；
- 关键架构不变量；
- 发布阻塞标准。

以后任何重构只更新一份基线，而不是在 README、PRD、审查报告中留下多个互相冲突的数字。

## 5. 阶段四：先建立书籍骨架和最小纵向闭环

### 5.1 不要一开始写满所有章节

先创建：

- 目录与导航；
- 术语表；
- 参考文献与代码索引；
- 每章学习目标和职责摘要；
- 一个最小可运行案例；
- 基础构建和链接检查。

然后沿一条最短路径完成纵向闭环：

```text
用户输入
→ 核心对象
→ 最小控制循环
→ 一项外部能力
→ 可观察结果
→ 成功或失败终态
```

这条闭环让后续章节有真实落点，也能尽早发现概念模型是否可实现。

### 5.2 最小案例的边界

最小案例应：

- 离线可运行或依赖极少；
- 明确哪些能力是模拟的；
- 至少包含一个失败路径；
- 不提前实现后面十章的所有组件；
- 不使用“生产级”“完整系统”等标签。

### 5.3 可复用提示词

```text
请根据现行 BookSpec 先建立书籍骨架，不要一次写满正文。
需要生成：目录、章节职责、术语表结构、引用结构、代码索引，
并实现一条最小纵向切片，证明【用户输入】可以经过【核心流程】得到可验证结果。
请明确教学边界、失败路径和后续章节尚未实现的能力。
```

## 6. 阶段五：根据需求正式写书

### 6.1 每个核心知识点采用决策叙事

不必机械使用五个固定标题，但读者最终应能找到：

| 维度 | 要回答的问题 |
|---|---|
| Why | 为什么需要它？解决什么失败模式？ |
| What | 它是什么？不是什么？ |
| How | 最小机制、接口和数据流是什么？ |
| When | 什么时候使用？什么时候不使用？ |
| Trade-off | 成本、复杂度、安全和维护代价是什么？ |

### 6.2 按章节类型采用不同结构

| 章节类型 | 重点 |
|---|---|
| 概念/历史 | 背景、边界、关系和证据 |
| 实现/工程 | 架构、接口、代码、失败与验证 |
| 协议 | 版本、角色、生命周期、标准字段、模拟边界 |
| 产品/框架 | 比较维度、公开事实、推导边界、时效性 |
| FAQ/汇总 | 指向权威章节，不重复整段正文 |

### 6.3 一章一章写，写完立即验证

推荐循环：

```text
明确本章职责
→ 写概念和边界
→ 添加必要图表
→ 添加代码或案例
→ 运行示例
→ 检查与前后章衔接
→ 更新索引和引用
```

不要等 20 章全部完成后才第一次运行代码或渲染图表。

---

## 7. 阶段六：建立自动化质量基线

完整技术书应该像软件项目一样接受持续验证。

### 7.1 推荐检查

- 文档生成器严格构建；
- Markdown 内部链接；
- 外部链接定期检查；
- Mermaid 或其他图表语法；
- 代码 Fence 语法；
- 每个示例工程构建；
- 自动测试；
- 格式和空白错误；
- 旧名称、旧路径、TODO、占位符搜索。

### 7.2 CI 分层

```text
快速检查：格式、链接、图表语法
→ 语言检查：Python/TypeScript/其他语言
→ 文档构建
→ 定期任务：外部链接与产品事实复核
```

### 7.3 验证结果也要写入规范

BookSpec 的 Current Baseline 应记录最近一次验证结果，但不要把本地一次成功描述为所有平台永久兼容。

---

## 8. 阶段七：多轮专项审查和优化

写完初稿后，不要只进行一次“综合审查”。综合审查往往只能发现表面问题。推荐分轮进行，每轮只处理一种缺陷。

### 8.1 第一轮：事实与术语

- 当前协议版本是否正确？
- 产品能力是否来自直接官方文档？
- 闭源行为是否被误写成内部实现？
- 是否区分标准、产品约定、本书抽象和新兴术语？
- 同一术语是否只有一个规范定义？

### 8.2 第二轮：章节结构和衔接

- 每章是否承担明确职责？
- 概念首次出现的位置是否合理？
- 后续章节是否重复完整定义？
- 第 N 章是否依赖尚未介绍的第 N+5 章？
- 综合实现章是否偷偷引入大量新概念？

### 8.3 第三轮：代码与正文

- 类名、方法名、路径和命令是否真实存在？
- 代码是否实现正文承诺？
- 输出是否来自实际运行？
- 失败、超时、取消和拒绝路径是否存在？
- “教学实现”“模式骨架”“生产要求”是否明确区分？

### 8.4 第四轮：图表与表格

- 图中箭头表示调用、依赖、包含还是数据流？
- 是否混合不同抽象层级？
- 状态图是否覆盖真实代码状态？
- 图号、标题、索引和正文引用是否一致？
- 表格是否真正承担比较作用？

### 8.5 第五轮：安全与治理

- 模型生成内容是否被当作可信命令？
- 写操作是否经过 Policy、Approval 和隔离边界？
- 安装是否被误认为授权？
- 凭据是否进入 Prompt、日志或 Trace？
- 超时是否被误写成底层副作用已取消？

### 8.6 第六轮：可读性与删减

- 删除重复定义、口号和无信息增量段落；
- 合并相似 FAQ 和表格；
- 缩短过长示例；
- 保留有职责的重复：代码映射、安全差异、选型比较；
- 不进行全书机械替换或无判断批量删段。

---

## 9. 阶段八：从按章审查转向纵向架构审查

多次重构后，最大的风险往往不是某一章写错，而是同一个概念在不同章节发生漂移。

### 9.1 建立 5～8 条贯穿全书的主线

通用示例：

```text
执行主线：请求 → 任务 → 执行 → 外部能力 → 结果
状态主线：会话 → 状态 → 检查点 → 恢复 → 观测
扩展主线：安装 → 启用 → 发现 → 加载 → 调用 → 卸载
协作主线：父主体 → 委派 → 子主体 → 结构化结果
治理主线：身份 → 策略 → 审批 → 隔离 → 审计
工程主线：输入设计 → 上下文 → 工具 → 工作流 → 运行环境 → 反馈循环
```

不同主题可替换为自己的主线，例如数据库书可以使用查询、事务、存储、复制、安全和运维。

### 9.2 纵向审查如何执行

对每条主线：

1. 找到首次定义章节；
2. 搜索全书所有相关术语；
3. 画出当前对象和状态关系；
4. 对照代码和图表；
5. 修正定义漂移；
6. 建立唯一权威章节和交叉引用；
7. 记录仍需取舍的 P2。

### 9.3 缺陷分级

| 等级 | 含义 | 发布要求 |
|---|---|---|
| P0 | 安全问题、严重事实错误、破坏性行为 | 必须修复 |
| P1 | 核心概念冲突、代码不可运行、图文生命周期错误 | 必须清零 |
| P2 | 适配器缺口、衔接问题、重要重复或遗漏 | 修复或记录取舍 |
| P3 | 措辞、排版和局部可读性 | 有选择地处理 |

---

## 10. 阶段九：架构和实现细节修订

审查发现问题后，修订顺序应从系统不变量到局部文字：

1. 身份和生命周期；
2. 架构职责和边界；
3. 数据结构与状态机；
4. 安全和失败语义；
5. Adapter 与扩展接入；
6. 代码和测试；
7. 图表和表格；
8. 章节衔接；
9. 措辞和排版。

如果先润色文字，再修改底层架构，之前的润色通常会全部失效。

### 10.1 复杂示例应该拆分

综合实现不必全部放在一个大文件中。建议按职责拆分：

```text
final-example/
├── application.*       # 会话和请求入口
├── assembly.*          # Port、Runtime、Host
├── governance.*        # Policy、Approval、Snapshot
├── adapters.*          # 安装面和外部系统桥接
├── scenario.*          # 具体业务场景
└── test.*              # 契约和端到端测试
```

拆分的判断标准不是行数本身，而是职责能否独立解释、测试和替换。

### 10.2 需求不合理时如何取舍

作者不需要预先具备所有背景。遇到不确定要求时，按以下证据排序：

```text
官方规范与源码
→ 官方工程文档
→ 成熟开源实现
→ 论文
→ 多个产品可观察到的共同模式
→ 明确标注的工程推导
```

不要因为最初需求写了某个结构，就强行保留已经被证据否定的设计。

---

## 11. 阶段十：最终反向修订需求文档

这是最容易遗漏、也最重要的一步。书籍经过实现和审查后，初始 PRD 几乎一定已经部分过时。

### 11.1 反向核对清单

- 章节编号和名称是否仍一致？
- 最终案例是否换了类名、目录或职责？
- 示例、图表和测试数量是否更新？
- 新增的核心概念是否进入 Coverage Matrix？
- 删除或降级的需求是否仍被写成阻塞项？
- 协议版本和产品核查日期是否更新？
- 旧 V1/V6/V7 是否仍看起来像现行规范？
- 项目结构是否与 GitHub 仓库真实目录一致？

### 11.2 历史需求的正确处理

不要悄悄改写历史，使其失去追溯价值。推荐：

- 保留原文；
- 在文件顶部加“已废止/仅作历史记录”；
- 链接当前 BookSpec；
- 在 PRD 根目录说明权威层级；
- 当前动态指标只维护在 Current Baseline。

### 11.3 最终冻结规则

- P0/P1 为 0；
- P2 已修复或记录取舍；
- 自动化检查全部通过；
- 不再新增外围模块；
- 冻结后只接受事实、安全、可运行性、图文一致性和必要出版润色修订。

.

├── examples/          # 独立可运行示例
├── specs/             # 现行 BookSpec、共创方法与历史需求
│   ├── book/
│   └── history/

├── .github/workflows/ # CI

├── CONTRIBUTING.md

根目录只保留 GitHub 访问者需要的入口。构建产物、依赖目录、缓存和会话临时文件通过 `.gitignore` 排除。

## 13. 作者与 AI 的协作方式

### 13.1 作者负责什么

- 提出主题、兴趣和真实目标；
- 对书籍定位和重大取舍作最终决定；
- 提供领域经验、案例和个人判断；
- 判断文字是否符合想表达的观点；
- 对公开发布和许可证负责。

### 13.2 AI 适合负责什么

- 将模糊目标转为可验收规范；
- 搜索跨章冲突、旧路径和重复内容；
- 参考官方资料和开源实现进行架构取舍；
- 编写、拆分和测试教学代码；
- 生成并校验图表、索引和交叉引用；
- 维护审查记录和 Current Baseline；
- 执行机械但需要一致性的全仓库修订。

### 13.3 AI 不应自行决定什么

- 未经确认改变书籍核心立场；
- 把无法验证的产品行为写成事实；
- 为了“看起来完整”新增不必要章节；
- 用大批量替换掩盖语义差异；
- 自动发布、提交或推送，除非作者明确授权；
- 删除历史依据或作者内容而不先判断其追溯价值。

### 13.4 高效协作节奏

推荐使用短指令推进，但每次都指定当前阶段：

```text
“先只审查，不修改。”
“按 P0/P1/P2 排序并修复 P0/P1。”
“继续实现第一个 P1，并同步文档和测试。”
“按六条纵向主线终审。”
“把确定性 P2 修复，不必要的记录取舍。”
“反向更新 PRD，不要让历史 PRD 覆盖当前实现。”
“整理 GitHub 结构，保留可追溯资料，删除生成物。”
```

作者不需要一次说出完美需求。更重要的是持续说明目标、指出不理解或不确定的部分，并允许证据推动架构调整。

## 14. 可直接复用的阶段提示词

### 14.1 需求升级

```text
请审查当前技术书 PRD。不要立即写正文。
识别定位、读者、学习成果、章节职责、最终案例、事实来源、代码和发布验收中的缺口。
将历史愿望清单升级为可执行 BookSpec，并建立规范优先级和 Current Baseline。
```

### 14.2 正式写作

```text
请严格根据现行 BookSpec 编写【章节】。
核心知识点需要覆盖 Why、What、How、When、Trade-off，
但不要机械套用固定标题。代码、图表和表格只在承担解释职责时加入。
产品与协议事实使用直接官方来源，并标明事实、推导和教学实现边界。
```

### 14.3 代码和正文一致性审查

```text
请核对【章节】的正文、图表、代码和测试。
检查类名、方法、路径、状态、失败语义、输出和运行命令是否一致。
先报告 P0-P3，再直接修复确定性的 P0/P1；不确定项依据官方规范、开源实现或明确推导取舍。
```

### 14.4 全书纵向终审

```text
不要再按章节孤立审查。请建立贯穿全书的执行、状态、扩展、协作、治理和工程实践主线，
搜索每条主线在所有章节、图表、术语表、FAQ 和代码中的定义漂移。
修复确定性冲突，生成终审报告，并记录剩余 P2 的取舍。
```

### 14.5 最终 PRD 修订

```text
书籍已经经过多轮重构。请以当前代码、目录、测试和书稿为事实基线，反向审查 PRD。
更新现行 BookSpec 的章节、概念、能力、数量和验收标准；
历史 PRD 保留原文但明确标记废止，不能继续覆盖当前实现。
```

## 15. 最终复用 Checklist

### 需求阶段

- [ ] 有明确读者、非目标和最终可验证成果
- [ ] 已区分已知可验证、已知待复核、推测相关、已知未知和潜在未知
- [ ] 不完全熟悉的领域已完成标准、开源实现、产品、论文和真实任务的证据扫描
- [ ] 核心成果已选择 L1～L7 层级，并至少为关键成果写出 Given/When/Then/Failure/Boundary/Evidence
- [ ] 已建立代表性任务集，而不只依赖 Happy Path
- [ ] 已用知识依赖图生成章节，并标记 Committed/Provisional/Backlog/Appendix/Rejected
- [ ] 已用生命周期 × 工程质量矩阵扫描可能遗漏
- [ ] 章节按学习路径组织，而不是概念堆叠
- [ ] 建立 PRD、BookSpec 和历史需求的权威层级

### 写作阶段

- [ ] 先完成最小纵向闭环，再扩展覆盖
- [ ] 核心知识点能回答 Why/What/How/When/Trade-off
- [ ] 事实、推导、观点和教学实现明确区分
- [ ] 每章写完立即运行相关验证

### 审查阶段

- [ ] 事实、术语、结构、代码、图表、安全分别审查
- [ ] 建立跨章纵向主线
- [ ] P0/P1 清零，P2 修复或记录取舍
- [ ] 删除无信息重复，保留有职责的重复

### 冻结阶段

- [ ] 正文、代码、图表、索引、FAQ 和 PRD 同步
- [ ] Current Baseline 与实际仓库一致
- [ ] 自动化检查全部通过
- [ ] 历史需求不会被误认为当前规范
- [ ] 仓库结构适合公开发布，不包含缓存、构建产物和临时记录

---

## 16. 一句话总结

写一本完整技术书，不是让 AI 连续生成很多 Markdown，而是把“需求、知识结构、实现、证据、测试、审查和维护”组织成一个可迭代、可验证、可冻结的工程系统；作者提供目标与判断，AI 提供结构化推演、跨文件一致性和高频验证，两者共同让书稿从愿望逐步收敛为可信的技术作品。
