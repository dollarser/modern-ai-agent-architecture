## 一、结论摘要

第 16 章当前实现适合作为**教学型 Reference Architecture**：它已经形成可运行的端到端闭环，采用 Port / Adapter、Composition Root、默认拒绝审批、结构化失败、Checkpoint、Hook、Event Bus、Handoff、Skill、MCP 和 Plugin 等合理构件，并且 Python / TypeScript 保持了较高程度的契约对等。

它还不能被称为生产级“最佳实践实现”。主要差距不在功能数量，而在扩展生命周期、恢复一致性、统一授权、作用域与并发隔离。尤其是扩展初始化失败后可能留下半加载状态，以及 Session 恢复时没有验证能力集合是否发生变化，这两项会直接影响正确性。

用户提出增加 Skill、MCP、Plugin 安装子系统的需求本身是合理的。OpenCode、Hermes Agent、Codex、Claude Code 和 GitHub Copilot 均公开支持其中若干组合；OpenClaw 也将 Skills 与工具策略作为扩展面的重要部分。需要调整的是实现边界：安装器负责供应链与落盘，Host 负责受控加载和生命周期，Policy Engine 负责统一授权，Checkpoint 负责绑定运行时能力快照。

建议保持第 12～14 章分别讲清 Skill、MCP、Plugin 的领域模型与安装管理，第 16 章只负责展示这三类扩展如何经由统一 Extension Runtime、Policy Engine 和 Capability Snapshot 被组装进 Agent Host。无需把所有安装细节堆进第 16 章。

## 二、当前实现已经做对的部分

1. **架构定位基本正确**：第 16 章明确定位为最终组装章，而不是重新实现所有子系统。
2. **依赖可替换**：`DatabaseReviewAgent` 通过构造参数注入 LLM、Memory、Hooks、Approval、MCP、Event Bus、Subagent 和 CheckpointStore。
3. **安全默认值合理**：默认使用 `DenyApprovalGate`，成功演示才显式注入自动审批。
4. **副作用前审批顺序正确**：Tool 支持 `prepare → approve → checkpoint → execute`，拒绝时 Handler 不执行。
5. **失败语义较诚实**：顶层异常会转为结构化失败，Session/Task 不匹配也不再误报成功。
6. **扩展来源已有雏形**：Tool 带来源信息，Plugin 有 Manifest/权限声明，Skill 有 Owner，MCP 有 Server Name。
7. **双语言契约对等**：Python 与 TypeScript 实现了相近的 Port 和运行流程，便于教学对照。
8. **离线可验证**：默认 Adapter 不依赖 API Key，适合书稿示例和 CI。

这些优点说明当前方向无需推翻。后续应把生命周期和一致性边界从“示例约定”升级为显式抽象。

## 三、问题清单与源码依据

### P0-1：扩展初始化不是事务，关闭也不是 best-effort（已修复）

> 2026-07-12 修复：Python 增加初始化锁，TypeScript 增加 single-flight Promise；双语言均实现逆序回滚、可重试初始化、best-effort 关闭和错误聚合，并加入失败注入回归测试。

**影响**：一次初始化失败可能留下部分 Skill、MCP Tool 或 Plugin。由于 `_extensions_initialized/extensionsInitialized` 仍为 `false`，下次重试会再次注册已有能力并触发冲突。关闭过程中若某个 Plugin 卸载失败，后续 Plugin、MCP 连接和 Tool 清理可能全部被跳过。

**Python 依据**：

- `DatabaseReviewAgent._initialize_extensions()` 依次执行 installed Skills、默认 MCP、managed MCP、installed Plugins，最后才设置 `_extensions_initialized = True`。
- 方法中没有 preflight、staging、commit 或异常回滚。
- `self._managed_mcp.append(adapter)` 和 `self._installed_plugin_names.append(...)` 在流程中逐步修改共享状态。
- `DatabaseReviewAgent.close_extensions()` 顺序卸载；任一 `plugins.unload()` 或 `mcp_servers.close()` 抛错都会中断后续清理。
- 审查时对应位置约为 `examples/agent-host/python/assembly.py:1030-1058`。

**TypeScript 依据**：

- `initializeExtensions()` 同样逐项修改 Registry 与数组，最后才写入 `extensionsInitialized = true`，没有回滚。
- `closeExtensions()` 没有逐项捕获和聚合清理错误。
- 审查时对应位置约为 `examples/agent-host/typescript/assembly.ts:976-998`。

**建议**：新增 `ExtensionRuntime`，采用 `discover → validate/preflight → stage → commit`。初始化失败按逆序 rollback；关闭采用 best-effort，继续清理全部资源并在最后返回/抛出聚合错误。初始化与关闭都应幂等，并由互斥锁保护。

**验收条件**：

- 任一 Skill/MCP/Plugin 在加载阶段失败后，Registry 与连接集合恢复到初始化前状态。
- 再次初始化不会因上次残留产生冲突。
- 一个 Plugin 卸载失败不影响其余 Plugin 和 MCP 清理。
- Python/TypeScript 均有失败注入与重试测试。

### P0-2：Checkpoint 缺少 Capability Snapshot，恢复可能改变语义（已修复教学契约）

> 2026-07-12 修复：Checkpoint 已保存 Tool/Skill/Policy/关键配置摘要和 `snapshot_hash`；不兼容恢复返回 `capability_snapshot_mismatch`，审批 ID 绑定快照 Hash。模型版本、MCP 协议身份与兼容迁移仍属于后续生产增强。

**影响**：计划或审批生成后，如果 Tool Schema、Skill/Plugin 内容、MCP Server、模型、配置或策略发生更新，同一 Checkpoint 恢复后可能执行与原计划不同的能力。已有审批也可能被错误复用到新版本 Tool。

**源码依据**：

- Python `RunState` 只保存 `session_id`、`task`、`plan`、`results`、`memory`、`trace`、`approvals`、计数、状态和错误；`to_dict()/from_dict()` 没有能力版本字段。审查时约为 `assembly.py:46-91`。
- TypeScript `RunState/StoredState` 同样没有能力快照。
- Approval ID 主要由 `session + step + tool` 组成，未绑定 Tool Schema/实现版本或策略版本。

**建议**：Checkpoint 增加不可变的 `CapabilitySnapshot`，至少包含：

- Tool：规范化名称、Schema 摘要、实现/来源版本、来源标识；
- Skill/Plugin：名称、版本、内容 checksum、来源与信任级别；
- MCP：Server ID、连接类型、协议版本、Tool Schema 摘要；
- Runtime：planner/model 标识、关键 AgentConfig 摘要、Policy 版本；
- 整体 `snapshot_hash`。

恢复时比较当前快照与保存快照。默认应拒绝不兼容恢复；只允许明确的兼容迁移或重新规划/重新审批。

**验收条件**：Tool Schema、Plugin checksum 或 Policy 版本变化会触发结构化 `capability_snapshot_mismatch`，不会静默继续；审批 ID 绑定 snapshot hash。

### P0-3：权限模型分散，缺少统一 Policy Engine（运行时已收敛，安装治理待深化）

> 2026-07-12 修复：新增独立 `governance.py/governance.ts`，运行时 Tool 授权统一为 `allow/ask/deny`，Allowlist 不再以并行 Guard 重复判断，ApprovalGate 只处理 `ask`。Plugin 安装权限和未来 managed/project/user 作用域仍需在后续 P1 阶段接入同一策略模型。

**影响**：当前授权逻辑分别存在于 Plugin Manifest 权限、Tool `requires_approval`、`ApprovalGate`、`before_tool` Guard 和 `allowed_tools/allowedTools`。多个入口可能得出不一致结论，也难以表达资源范围、参数条件、配置作用域和企业强制策略。

**源码依据**：

- `PluginManifest.permissions` 负责扩展加载权限。
- `Tool.requires_approval/requiresApproval` 决定是否进入人工审批。
- `ApprovalGate` 只返回布尔决定。
- `_enforce_allowlist()` / `before_tool` Guard 使用全局 Tool 名集合。
- `AgentConfig.allowed_tools/allowedTools` 与扩展安装器自己的权限检查相互独立。

**建议**：建立统一决策接口：

```text
PolicyDecision evaluate(
  subject, capability, arguments, resource,
  config_scope, session, provenance, risk
) -> allow | ask | deny
```

`ApprovalGate` 退化为 `ask` 决策的交互 Adapter，而不是独立安全策略。所有 Tool、MCP、Skill 内容加载、Plugin Hook/Command 注册和安装/更新操作都经过同一策略语义。合并规则应使更高层级的 `deny` 无法被低层配置放宽。

**验收条件**：同一请求只产生一个可审计的最终决定；决定包含命中规则、策略版本、作用域和理由；默认拒绝未知能力。

### P1-1：DatabaseReviewAgent 同时承担 Host、Composition Root 和业务 Demo（已修复）

> 2026-07-12 修复：新增通用 `AgentHost`；数据库审查 Skill、Tool、Plugin、Handoff Handler 和 Memory Seed 已移入独立双语言 Scenario 模块。具体应用明确命名为 `DatabaseReviewAgent`，不保留旧兼容类型。结构测试确认裸 Host 不包含业务能力。

**影响**：通用运行时与“数据库配置审查”示例耦合，导致 Host 难以复用，步骤结果还通过数字索引相互引用，计划一旦调整容易破坏 Handler。

**源码依据**：

- Python `_assemble()` 注册 `config-review`、`inspect_candidate`、`delegate_review`、`compose_report` 和 `review-pack`，并在 Handler 中读取 `ctx.results[1]`、`[2]`、`[3]`、`[4]`、`[5]`；审查时约为 `assembly.py:919-967`。
- `_run()` 写入“数据库配置变更必须给出文件路径并经过审查”的默认 Memory；审查时约为 `assembly.py:1063-1069`。
- TypeScript `assemble()` 与 `runInternal()` 存在同样耦合，审查时约为 `assembly.ts:855-930`、`1005-1011`。

**建议**：把通用 `AgentHost`、Extension Runtime、Execution Runtime 留在 assembly/runtime 模块；将数据库配置审查移入 `demo.py/main.py` 与 `demo.ts/main.ts`，通过 `ApplicationBundle` 或工厂函数注入。步骤依赖优先使用命名输出/显式输入映射，不直接依赖固定数字索引。

### P1-2：缺少 user/project/local/managed/session 作用域与配置优先级（策略作用域已修复）

> 2026-07-12 修复：双语言治理模块新增 `ConfigScope`、`PolicyLayer` 和限制性合并；managed/project/user/local/session 均可表达，任何层的 deny 都不可被 allow 放宽。普通非安全配置的逐字段来源追踪仍可继续深化。

**影响**：真实开发工具需要区分个人配置、项目共享配置、本机私有配置、企业托管配置和单次 Session 覆盖。当前 Provider 只有“已安装/已启用”概念，无法清楚处理来源、冲突和管理员强制限制。

**建议**：定义统一 `ConfigScope` 与来源优先级。一个可行模型为：

```text
managed policy（只收紧）
  → project shared
  → user
  → local private
  → session override（不得突破 managed deny）
```

精确优先级可按产品目标调整，但必须显式记录最终值的来源。权限合并不应使用普通“后者覆盖前者”，而应采用 deny 优先的限制性合并。

### P1-3：Skill 不是完整的渐进式披露（已修复教学契约）

> 2026-07-12 修复：`SkillRegistry` 拆分 `discover()` 与 `load()`；发现阶段只返回 Metadata，选择后才加载正文，并记录 `skills_discovered/skills_loaded` Trace。内容 Token 预算和信任扫描仍由生产 Adapter 扩展。

**影响**：`SkillRegistry.match()` 通过关键词匹配后，运行流程直接把匹配 Skill 的完整 `instructions` 注入 Context。Skill 数量增长时会消耗上下文，也缺少读取前授权、信任与审计。

**源码依据**：

- Python `SkillRegistry.match()` 返回完整 `Skill`；`_run()` 在规划前使用匹配结果。审查时约为 `assembly.py:326-350`、`1099` 之后。
- TypeScript `match()` 后直接 `map(skill => skill.instructions)`；审查时约为 `assembly.ts:299-315`、`1050`。

**建议**：拆分 `SkillMetadata` 与 `SkillBody`：先发现元数据与摘要，再排序和策略判断，最后按需加载正文，并记录 `skill.discovered/selected/loaded/rejected` 事件。支持内容预算和同名来源优先级。

### P1-4：MCP Tool 命名不稳定，协议生命周期仍是教学子集（命名已修复）

> 2026-07-12 修复：MCP Tool 内部名称始终为 `<server>.<tool>`，短名仅作为显式、无歧义 Alias；Alias 映射进入能力快照，安装顺序不再改变 canonical identity。OAuth、动态 `list_changed`、重连和上下文预算仍属于后续协议深化。

**影响**：当前第一个无冲突 Tool 使用短名，只有碰撞发生时后来的 Tool 才使用 `<server>.<tool>`。安装顺序改变会改变可见名称，计划与 Allowlist 可能因此失效。实现也尚未覆盖生产常见的超时、重连、OAuth、动态 Tool 列表和上下文预算。

**源码依据**：

- Python `MCPToolAdapter.register_tools()` 先使用远端短名，仅在 `registry.contains(name)` 时加 Server Namespace；审查时约为 `assembly.py:398-423`。
- TypeScript `registerTools()` 使用相同策略；审查时约为 `assembly.ts:348-370`。

**建议**：内部 canonical name 始终使用 `<server-id>.<tool>`，短名仅作为经过歧义检测的显示别名。补充连接超时、调用超时、取消、重连退避、OAuth/凭证引用、`list_changed` 刷新、Schema 变更处理、采样/elicitation 策略和 Tool 上下文预算。Tool 多时应支持延迟发现或 Tool Search，而不是一次性全部注入模型上下文。

### P1-5：共享 Registry 和初始化标记缺少并发保护（核心风险已修复）

> 2026-07-12 修复：初始化已由 Python Lock / TypeScript single-flight 保护；每次 Run 在规划前复制冻结 Tool Registry，执行器只使用该视图，运行中 Registry 替换不会改变已有计划。跨进程能力租约与连接延迟释放仍属于生产增强。

**影响**：两个 Session 并发首次运行时可能同时初始化扩展；运行中的计划也可能受到热加载/卸载影响。`_memory_initialized` 与 `_extensions_initialized` 只是布尔值，不构成同步原语。

**源码依据**：Python/TypeScript 的 Registry、`managedMcp`、Plugin 名单和初始化标记都属于可变共享状态，初始化入口没有 lock/single-flight。

**建议**：

- 初始化与关闭使用 lock 或 single-flight Promise；
- 每次 Run 在规划前捕获不可变的 per-run Capability Snapshot；
- Registry 更新采用 copy-on-write/版本化视图；
- 热卸载只影响新 Run，已有 Run 持有的能力租约结束后再真正释放连接。

- 依据：[Repository](https://github.com/nousresearch/hermes-agent)、[Skills](https://hermes-agent.nousresearch.com/docs/user-guide/features/skills)、[Plugins](https://hermes-agent.nousresearch.com/docs/user-guide/features/plugins)、[MCP](https://hermes-agent.nousresearch.com/docs/user-guide/features/mcp)。

### 4.4 OpenAI Codex

- 官方文档公开支持 stdio 与 Streamable HTTP MCP、Bearer/OAuth、共享配置和项目作用域。
- Skills 是可发现的扩展单元；安全文档强调 sandbox、approval 与受控权限边界。
- 依据：[MCP](https://developers.openai.com/codex/mcp/)、[Skills](https://developers.openai.com/codex/skills/)、[Security](https://developers.openai.com/codex/security/)、[Configuration](https://developers.openai.com/codex/config-basic/)。

### 4.5 Claude Code

- Plugin 可组合 Skills、Agents、Hooks、MCP、LSP 等能力，并支持 user/project/local/managed 等安装或配置范围。
- Marketplace、权限与 Sandbox 文档体现供应链、作用域、权限和运行隔离是不同层次。

### 4.6 GitHub Copilot

- 官方文档公开了 Agent Skills、MCP 管理、Hooks 和企业策略等扩展/治理能力。
- 云端 Coding Agent 运行在临时、受限制环境中，说明扩展配置不能替代执行隔离。
- 不同 Hook 的失败行为可能不同，因此 Guard 与 Observer 的失败语义必须显式设计，不能笼统处理。
- 依据：[Agent Skills](https://docs.github.com/en/copilot/how-tos/copilot-on-github/customize-copilot/customize-cloud-agent/add-skills)、[MCP management](https://docs.github.com/en/copilot/concepts/mcp-management)、[Hooks reference](https://docs.github.com/en/copilot/reference/hooks-reference)、[Agents](https://docs.github.com/en/copilot/concepts/agents)。

### 4.7 从对标中提炼的共性

1. Skill、MCP、Plugin 并非不合理的重复功能，而是不同抽象层的扩展面。
2. 安装来源、配置作用域、运行权限和执行隔离必须分层。
3. `allow / ask / deny` 比多个互不相干的布尔开关更适合作为统一授权语义。
4. Skill 应优先采用元数据发现与按需加载，MCP Tool 应控制上下文预算。
5. 企业/managed 策略只能收紧低层配置，不能被项目或用户配置绕过。
6. Plugin 是能力打包与生命周期单元，不应天然获得进程内完全信任。

---

## 五、建议修复顺序

### 第一阶段：正确性与可恢复性

1. 实现事务式 `ExtensionRuntime`，含初始化 lock、rollback、幂等 close 和聚合清理错误。
2. 为 RunState/Checkpoint 增加 `CapabilitySnapshot`，恢复时执行兼容性检查。
3. 建立统一 `PolicyEngine`，把 Allowlist、Approval、Plugin 权限和 MCP/Skill 策略收敛到 `allow/ask/deny`。

### 第二阶段：职责与配置治理

4. 将数据库配置审查业务 Demo 从通用 `DatabaseReviewAgent/AgentHost` 中抽离。
5. 增加 user/project/local/managed/session 作用域、来源追踪和限制性合并。
6. 为每个 Run 捕获不可变能力视图，处理并发初始化和热加载/卸载。

### 第三阶段：扩展面深化

7. Skill 改为 metadata discovery → rank → policy → body load → audit。
8. MCP 内部名称始终命名空间化，并补超时、重连、OAuth、动态列表和 Tool 预算。
9. Plugin 增加更清晰的隔离等级与宿主 API 版本契约。

## 六、当前不必优先实现的内容

以下功能有价值，但不是把第 16 章提升为可靠 Reference Architecture 的前置条件：

- 公共 Marketplace、评分和搜索；
- 任意 Git URL 自动下载；
- 完整 SemVer 求解器；
- 自动 Skill 学习/生成；
- 分层多 Agent 集群；
- 分布式 Event Bus；
- 向量数据库与生产级长期记忆；
- 多进程或容器级 Plugin Sandbox 的完整实现。

书稿可以说明这些生产演进方向，但不应为了“功能齐全”把教学实现扩大为难以验证的平台。

---

## 七、完成定义

完成上述优先修复后，第 16 章可以更准确地定位为：

> 一个具有事务式扩展生命周期、可验证恢复语义、统一授权决策、显式配置作用域和并发安全能力视图的教学型 Agent Host Reference Architecture；生产部署仍需替换持久化、隔离、密钥、身份、可观测性和分布式基础设施。

验证至少应覆盖：

- Python 与 TypeScript 扩展加载中途失败回滚；
- 重复初始化、并发初始化、重复关闭；
- 关闭时多错误聚合和后续资源继续释放；
- Capability Snapshot 相同/兼容/不兼容恢复；
- Tool 或 Policy 更新后旧审批不可复用；
- managed deny 不可被 project/user/session 放宽；
- Skill 正文按需加载且有预算与审计；
- MCP canonical name 在安装顺序改变后保持稳定；
- 业务 Demo 抽离后，通用 Host 不包含数据库审查专用 Tool、Memory 或结果索引假设；
- 全仓 Python/TypeScript、Markdown Fence、内部链接、Mermaid、MkDocs strict 与 `git diff --check` 回归通过。
