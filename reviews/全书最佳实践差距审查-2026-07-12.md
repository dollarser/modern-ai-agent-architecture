# 全书最佳实践差距审查与修复记录（2026-07-12）

> 范围：第 16 章之外的第 1～15、17～20 章。
> 方法：本地源码与书稿审查、跨章契约比对、开源仓库和产品官方文档核验。

## 结论

第 1～3、7、9～10、12、14、20 章整体方向正确，不需要结构性推翻。主要差距集中在“示例看似完整，但安全、身份、恢复或多租户契约没有进入代码”的章节。本轮已按风险顺序修复两项 P0、六项 P1 和若干一致性问题。

第二轮概念边界复审进一步补齐了四套横向分类：运行身份（Task/Run/Session/Checkpoint/Trace）、安全控制（Policy/Approval/Guardrail/Hook/Sandbox）、控制流（Workflow/Agent/Orchestrator）和管理组件（Catalog/Loader/Manager/Resolver/Registry/Router/Scheduler）。同时统一 Context/Observation/Artifact/Resource/Memory/Knowledge、身份与委派，以及 Skill/Rules/Command、App/Connector/Adapter、MCP/A2A/ACP 的边界。

第三轮重构后一致性审计修复了新旧内容漂移：第 16 章双语言实现将误作执行键的 `session_id/sessionId` 迁移为 `run_id/runId`，应用级 Session 明确由上层组合多个 Task/Run；第 1、2 章旧概念图按 Host/运行主体/能力边界重画；第 18 章 Hook 失败语义与 Guard fail-closed 对齐；7 张图统一为图后说明；TypeScript 示例索引改为按 `build/test/start` 实际脚本运行；Memory 检索和长任务流式表述改为混合检索与状态/进度/Artifact 契约。A2A 用于支撑 Context 与 Task 的分离；OpenTelemetry 仅作为通用 Trace/Span 语义参考，不作为 Agent 架构或运行身份依据。

第四轮将 Prompt、Context、Tool、Workflow、Harness、Loop 与 Agentic Engineering 纳入全书导航。取舍不是“六阶段演进”，而是正交且逐步扩大的工程关注面：第 1 章做概览，第 2 章做章节路由，第 4 章定义 Context Engineering，第 16 章将 Harness Engineering 映射到 Agent Host Port，第 18 章定义跨 Run Loop Specification、反模式与 Agentic Engineering 总框架。Harness 依据 OpenAI 公开工程文章，Workflow/Agent 依据 Anthropic 公开文章；Loop Engineering 明确标注为尚无跨产品统一标准的新兴术语。

# 五类正交概念补充审查（2026-07-12）

本轮在原有审查基础上进一步统一全书的概念架构：

- 用 `Application / Agent Host` 作为组合与治理边界，明确其不等于 Agent。
- 将 Agent/Subagent、Tool/Skill、MCP、Connector、Plugin 分别归入运行主体、能力、协议、产品集成和分发单元；ExpertProfile 单列为角色配置。
- 增加“安装 → 启用 → 发现 → 装载 → 调用 → 观测”治理链，明确安装成功不代表本次运行获权。
- 增加 Skill 脚本安全执行链，禁止 Skill Loader 绕过 Runtime 直接执行脚本。
- 补充 Connector 最小契约、健康/撤销生命周期与凭据引用边界。
- 统一 `Plugin Tool` 为 `Plugin-contributed Tool` 的来源表述，明确 Plugin 不是 Tool。

主要依据为 MCP 官方 Architecture/Authorization/Versioning、Agent Skills 规范与 OpenAI Skills 仓库，以及 OpenClaw、Hermes Agent 等公开仓库中对 Tool、Skill、Plugin、MCP 和 Subagent 的实现边界。产品名称只作为实例，不据此推断未公开的内部架构。
